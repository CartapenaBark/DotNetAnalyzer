## 报告漏洞

如果您发现安全漏洞，**请不要公开提交 issue**。相反，请按照以下步骤操作：

### 私密报告漏洞

1. **发送邮件**至项目维护者：paradox0520@outlook.com
2. **在邮件主题中注明** `[SECURITY]` - 漏洞简要描述
3. **在邮件正文中包含**：
   - 漏洞的详细描述
   - 受影响的版本
   - 重现步骤
   - 潜在影响
   - 如果可能，提供修复建议

### 我们会做什么

1. 在收到报告后的 **48 小时内**确认收到
2. 在 **7 天内**进行初步评估
3. 与您合作理解和解决漏洞
4. 在修复方案准备好后，协调发布日期
5. 在安全更新发布后，公开致谢（如果您同意）

### 我们不会做什么

1. 在未修复漏洞前公开披露漏洞
2. 未经您的同意公开您的身份
3. 忽略安全报告

## 安全最佳实践

### 作为用户

1. **保持更新**：始终使用最新版本的 DotNetAnalyzer
2. **审查代码**：在处理敏感代码库时，确保您信任 MCP 服务器配置
3. **网络隔离**：如果处理高度敏感的代码，考虑在隔离环境中运行
4. **日志监控**：定期检查 MCP 服务器的访问日志

### 作为开发者

1. **依赖审计**：定期运行 `dotnet list package --vulnerable` 检查依赖漏洞
2. **代码审查**：所有代码更改必须经过审查
3. **安全测试**：使用 SAST/DAST 工具进行安全扫描
4. **最小权限**：MCP 工具应仅请求必要的权限

## 已知的安全考虑

### 代码分析安全

DotNetAnalyzer 作为 MCP 服务器，具有访问和分析代码的能力。请注意：

- **本地执行**：分析在本地环境中进行，代码不会发送到远程服务器
- **只读操作**：默认情况下，分析工具是只读的，不会修改代码
- **显式确认**：重构和代码修改操作需要用户明确确认

### 敏感信息处理

- **不收集**：我们不会收集您的代码或分析结果
- **不传输**：除了 MCP 协议通信外，没有外部网络请求
- **本地存储**：所有分析数据存储在本地内存中，进程结束后清除

## 安全更新

### 如何获取安全更新

1. **关注 Releases**：在 GitHub 上 Watch 项目，选择 "Custom" -> "Releases only"
2. **订阅公告**：加入我们的邮件列表或关注社交媒体
3. **定期检查**：运行 `dotnet tool update -g dotnet-analyzer`（如果安装为全局工具）

### 验证更新

所有 NuGet 包都经过签名验证。您可以通过以下方式验证：

```bash
dotnet nuget verify DotNetAnalyzer.Core.x.y.z.nupkg
```

## 安全审计

我们欢迎安全研究人员对我们的代码进行审计。如果您在安全审计中发现问题，请按照上述"报告漏洞"部分的流程联系我们。

### 安全审计政策

- **授权审计**：对公开发布的代码进行审计是授权的
- **不要滥用**：不要对实时系统进行破坏性测试
- **负责任披露**：按照负责任的漏洞披露流程报告发现的问题
